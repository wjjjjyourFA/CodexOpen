## Warning

int32 等整型数据，注意在写入时，应当 x1e3 \ x1e2 \ x1e6 等放大，否则会丢失精度，导致误差累积。

因此在接收方使用时，应当 x1e-3 \ x1e-2 \ x1e-6 等缩小，恢复原始数据。

### Encoder.msg

直接由底盘发出，因此这里只设置 odometry.msg
  ==> 也就是 局部坐标系 local_pose



### LocalizationEstimate.msg

本质上就是一种融合后的导航信息数据结构，用于描述车辆在 **局部坐标系** 下的位置、速度、姿态等信息。
✅ GNSS（GPS）信息
✅ IMU（惯性测量单元）信息
✅ 码盘（里程计 / 编码器 / Dead Reckoning）信息
✅ 以及系统状态、姿态角、速度、协方差等估计结果
代表了 经过融合算法（如EKF、UKF、RTK、GNSS+INS等）处理后的“最终定位状态”输出。

**LocalizationEstimate.msg 是融合解算，不是定位数据打包。**


### Gps.msg 

由 GNSS GPS RTK 等得到，代表全局坐标系

**高斯坐标系**：

| 坐标系名称           | 长半轴 aaa | 扁率 fff          | 使用场景                     |
| -------------------- | ---------- | ----------------- | ---------------------------- |
| **北京54（BJ54）**   | 6378245.0  | 1 / 298.3         | 中国早期军用/测绘            |
| **西安80（Xian80）** | 6378140.0  | 1 / 298.257       | 80年代至21世纪初国内常用标准 |
| **WGS-84**           | 6378137.0  | 1 / 298.257223563 | ✅ GPS、北斗、国际通用标准    |

------

#### ✅ 实际工程/开发中通常用哪个？

> **绝大多数情况你应该使用 `WGS-84`。**

------

#### 📌 推荐使用场景：

| 场景                       | 推荐坐标系            | 说明                                           |
| -------------------------- | --------------------- | ---------------------------------------------- |
| GPS 定位                   | ✅ WGS-84              | 所有 GPS 卫星坐标、GNSS 通信统一标准           |
| 北斗定位（原始数据）       | ✅ WGS-84              | 北斗卫星坐标也是 WGS-84，只是结果有偏移（GCJ） |
| 地图服务（高德/腾讯/百度） | WGS84 → GCJ-02 → BD09 | 各自使用不同加密版本                           |
| 老旧军用或国土数据转换     | BJ54 / Xian80         | 若处理旧的军用地图，才使用 BJ54、Xian80        |
